# Cask 版本格式不要自创

**日期**: 2026-01-13 
**影响**: 版本更新方案反复偏离仓库现有写法，导致无效修改建议 
**发现人**: 用户

## 问题

更新 cask 版本号时，没有先检查仓库里已有写法，连续提出了逗号分隔、额外 `build` 变量这类不一致方案。

## 现象

最小复现是直接凭印象改版本格式，例如：

```bash
grep "version" Casks/*.rb
```

没有先执行这一步，就会提出 `0.2.0,52` 或独立 `build = "52"` 之类的写法。

## 根因

错误假设是“版本格式可以为了表达清晰随意调整”。实际约束是这个仓库已经有稳定风格，新增格式会让维护成本上升。

## 修复

先查看现有 cask 的 `version` 写法，再按仓库已有模式修改。本仓库当时应保持 `0.2.0.52` 这种点分格式。

## 预防

- 改任何 cask 前，先检查 `Casks/` 里的现有模式。
- 只在仓库已有先例的前提下调整版本格式。
- 不为局部表达方便引入新的字段或分隔规则。
